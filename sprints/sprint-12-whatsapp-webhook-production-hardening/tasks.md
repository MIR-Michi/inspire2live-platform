# Sprint 12 — Tasks

| ID | Task | Purpose / Scope | Acceptance Criteria | Owner | Status | Notes |
|---|---|---|---|---|---|---|
| S12-T01 | Parse and persist Meta `statuses` webhook events | Inbound webhook currently only handles `messages`; Meta also sends `statuses` change events (sent/delivered/read/failed) for messages this platform sent via the Graph API. Extend `parseWhatsAppWebhookPayload` / `processWhatsAppWebhookPayload` in `comms-webhook.ts` to recognise these and update the matching `whatsapp_outbound_messages` row (matched by `graph_message_id`). | New payload shape is parsed without breaking existing `messages` handling; matching outbound row's `delivery_status` updates to `delivered`/`read`/`failed` (+ `error_detail` on failure); unmatched status events are ignored without error; unit tests cover all status transitions plus the "no matching outbound row" case. | TBD | Not Started | Add `delivered_at` / `read_at` timestamp columns to `whatsapp_outbound_messages` via new migration if needed. |
| S12-T02 | Surface failed webhook events for operator triage | `whatsapp_webhook_events` rows with `processing_status = 'failed'` are currently invisible outside direct DB access. Add a small admin/comms-facing view (new page or section on an existing monitoring/intake page) listing failed events with `failure_reason`, `received_at`, and sender info. | Comms/admin users can see failed webhook events with reason and timestamp; RLS continues to restrict to `is_comms_team_or_admin()`; page/section has a test or is covered by existing access-control tests. | TBD | Not Started | Reuse existing `is_comms_team_or_admin()` policy; no new table needed — query `whatsapp_webhook_events` directly. |
| S12-T03 | Add a "replay" action for failed webhook events | Once failed events are visible (S12-T02), give operators a way to re-run `processWhatsAppWebhookPayload` against the stored `payload` for a single failed event without waiting for Meta to redeliver. | Server action re-processes a single `whatsapp_webhook_events.payload`, updates `processing_status`/`failure_reason` on success or failure, and revalidates the triage view; covered by a unit test using a stored failing payload fixture. | TBD | Not Started | Depends on S12-T02 for the UI entry point. |
| S12-T04 | Conversation threading between inbound and outbound messages | `intake_items` (inbound) and `whatsapp_outbound_messages` (outbound, via `in_reply_to_intake_item_id`) aren't visibly linked. Add a simple thread view — given an intake item or a WhatsApp contact (`sender_whatsapp_id` / `recipient_whatsapp_id`), show inbound + outbound messages in chronological order. | WhatsApp inbox / intake detail shows a chronological thread of inbound and outbound messages for a given WhatsApp ID; works with zero, one, and many prior messages; covered by a component or data-layer test. | TBD | Not Started | Likely a new data-loader in `comms-dashboard-data.ts`-style module joining the two tables by WhatsApp ID. |
| S12-T05 | WhatsApp webhook monitoring summary | No aggregate visibility into webhook health (volume, failure rate, last received). Add a summary (counts of accepted/duplicate/failed over a recent window, last `received_at`) to an existing monitoring surface or a new comms dashboard tile. | Summary reflects live counts from `whatsapp_webhook_events`; covered by a unit test for the aggregation query/helper; tile renders without errors when there are zero events. | TBD | Not Started | Check `/api/monitoring` (referenced in `docs/MONITORING.md`) for an existing pattern to extend before adding a new endpoint. |
| S12-T06 | Document WhatsApp Cloud API setup end to end | `.env.example` lists the required `WHATSAPP_*` variables but `docs/ENVIRONMENT_REFERENCE.md` omits them, and there's no operator walkthrough for the Meta App / webhook callback configuration. | `docs/ENVIRONMENT_REFERENCE.md` gets a "WhatsApp Cloud API" section matching the existing table format (variable, scope, required, default, description, "if missing" impact); new `docs/WHATSAPP_WEBHOOK_SETUP.md` walks through: creating/configuring the Meta App, setting the webhook callback URL + verify token, subscribing to `messages` and `statuses` fields, obtaining the access token and phone number ID, and verifying the integration end to end. | TBD | Not Started | Pure documentation — no code changes; can ship independently of other tasks. |
| S12-T07 | Regression coverage for the full webhook round trip | Existing tests cover webhook auth and classifier logic in isolation. Add coverage that exercises the full inbound → classify → intake → (S12-T01) status update flow using realistic Meta payload fixtures. | New test fixtures for `messages` and `statuses` payloads based on real Meta webhook shapes; integration-style test (or focused unit tests per stage) covers the round trip; `pnpm test` stays green. | TBD | Not Started | Can be split across S12-T01/T04 PRs rather than landing as one big test sprint-end task. |
